## Verification methodology

For L.5 and L.6, mirror L.3's research pattern:

1. Start from UNECE search (for L.5) or BOE search (for L.6) for each R-number / RD
2. Prefer modern deep-link patterns:
   - UNECE: `unece.org/transport/documents/YYYY/MM/standards/…` OR `fileadmin/DAM/trans/main/wp29/wp29regs/…` OR `sites/default/files/YYYY-MM/…`
   - BOE: `boe.es/buscar/act.php?id=BOE-A-YYYY-NNNNN`
3. For each URL — log revision label + retrieval date + brief evidence (what search returned, which result selected)
4. Never fabricate: if a deep link cannot be confirmed with ≥ 80% confidence, leave the rule at SEED_UNVERIFIED and note in L.5 README that it was held back

## Risks & rollback

| Risk | Mitigation |
|---|---|
| L.5 URL research yields < 12 confirmable deep links | Ship partial batch; the quality gate allows 12/20 worst case. Remaining rules roll to L.7. |
| Homologación Individual Orden ministerial citation remains `[verify]` | Keep REG-MS-ES-008 at SEED_UNVERIFIED; only -007 and -013 promote. |
| RD 106/2008 partially superseded | Verify current amendment status; if ambiguous, keep -013 at SEED_UNVERIFIED. |
| Pilot regression anchor shifts unexpectedly | Investigate each rule-ID diff before committing. `fixtures/pilot-my2027-bev.expected.ts` `applicable_rule_ids` may grow but must not shrink. |
| Test failures from new ACTIVE rules | Most tests are powered by the governance integrity checks (`activeWithoutUrl`, etc.) — if those fail, the specific rule is at fault, not the test. |

### Rollback
Each round is a single commit. `git revert <sha>` removes only that round's changes. The three rounds are independent.
